# Tool: edr_incident_view

用途：
- 查询单条 incident 的详情。
- 适合回答“这条事件具体发生了什么”“某个 incident_id 的详细上下文是什么”。

输入要求：
- 必须有 incident_id。
- 必须有 client_id。
- `incident_id` 只能来自两种来源：用户明确提供，或前面真实工具结果里已经返回的字段。
- 不要根据 `host_name`、`incident_name`、自然语言描述、时间、样例文本去猜测或拼接 `incident_id`。
- 如果只有主机名或事件标题，没有真实 `incident_id`，应先继续用 `edr_incidents` 缩小范围，而不是直接调用详情。

输出建议：
- 默认不要只贴原始字段；先把这条事件“讲明白”。
- 默认用纯文本自然说明，不要用 markdown 标题、加粗、代码块、表格，也尽量不要写机械的 1) 2) 3) 结构。
- 优先按这个顺序组织：这是什么事件 -> 发生在谁身上 -> 什么时候开始/结束 -> 中间出现了哪些关键行为 -> 为什么可疑/风险点在哪 -> 现在适合继续看什么。
- 如果详情里有时间线、IOA、进程链、启动项、计划任务、网络连接、注册表、文件落地等线索，要主动挑出最关键的几条串起来说明，不要只罗列字段名。
- 默认补一个简单进程链，用最短的人话串起来：入口进程 -> 子进程/载荷 -> 注入目标 -> 持久化/联网/横向动作；如果链不完整，就明确说目前只确认到哪一段。
- 默认补一个“事件 IOC”小段，只列真实出现过的文件名、路径、哈希、进程名、命令行、IP、域名、注册表项、计划任务、服务名、自启动项；没有就明确说当前还没提取到稳定 IOC。
- 如果用户说“随机看一个”“详细介绍一下”，默认给相对完整的事件解读，至少覆盖主机、核心行为链、关键证据、影响判断、建议下一步。
- 如果详情过大并转成 artifact，不要停在“已读某一段”；要基于读到的片段继续解释这段在整起事件里代表什么，必要时继续搜索或继续读，直到能给出像事件简报一样的回答。
- 如果辅助分析已经能支撑当前用户问题，就应该停止继续搜索，直接把结论整合成最终回答。
- 只有在辅助分析明确指出还有关键缺口，或者用户明确要求更深的字段/原文/证据时，才继续搜索或继续读。
- 如果需要分点，也优先用短横线或自然分段，不要出现 **事件概述**、**影响与风险** 这类 markdown 小标题。
- 不要向用户暴露内部分析状态，不要出现“辅助分析”“根据刚才的搜索结果”“enough_to_answer=true”“next_queries”这类内部术语。
- 如果用户明确说想看原始字段，可以再补原始详情，但原始字段应放在解释之后。
- 不要把列表接口的摘要当成详情，要明确区分。
